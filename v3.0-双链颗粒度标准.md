# v3.0 双链颗粒度与完整性标准

> **目的**：把双链从"单元级抽查"升级为"段落级全量+动态维护"。
> **覆盖缺口**：#11 双链只有单元级、#12 终检是抽查不是全量、#13 无动态双链机制、#14 无链接强度分级

---

## Part 1: 段落级双链标准（补缺口#11）

### 1.1 双链三级体系

v2.0只有文档级双链（CON-001 → OPI-003）。v3.0新增段落级和句子级：

| 级别 | 链接粒度 | 格式 | 适用场景 |
|------|---------|------|---------|
| **句子级** | 具体的一句话 → 具体的SRC位置 | `[原文]" — SRC-XX-003, 12:34` | 关键定义、核心证据、精确引用 |
| **段落级** | 一个段落 → 一段SRC范围 | `本段基于 SRC-XX-003, 12:34-15:00` | 概括性引用、背景描述 |
| **文档级** | 一个文档 → 多条SRC | frontmatter的 `source_documents` | 整篇文档的素材来源 |

### 1.2 知识库产物的段落级双链要求

每个 `.md` 文件必须满足：

```
1. 至少70%的段落有段落级或句子级溯源标记
2. 每个"此人认为/此人说/此人的观点是"类型的声明必须有句子级溯源（L3精确级）
3. 每个分析者概括/抽象必须有段落级溯源（基于哪段素材）
4. 没有溯源标记的段落 → 必须标注为"分析者综合推断"或"背景性描述"
```

### 1.3 段落级双链示例

```markdown
## CON-001: 价值交换

此人认为"价值交换"不是通常理解的"公平交易"。

[段1: 此人定义]
他把"价值交换"定义为"每个动作都在交换某种价值，即使你自己没意识到"。
这句话的核心是"意识不到"的部分——大多数人在交换价值时不知道自己交换了什么。
[本段基于: SRC-LDM-045, 12:34-15:00, "[原话: 你每次开口都在交换一个东西，只是你自己不知道交换的是什么]"]

[段2: 与通用理解的差异]
通用理解中"价值交换"是自愿的、明确的——我付钱你给东西。
但此人的定义包含了"无意识的、隐性的、被迫的"交换。
[本段基于: SRC-LDM-045, 15:00-18:30; SRC-LDM-078, 05:20-08:00; SRC-LDM-112, 22:00-25:00]

[段3: 操作公式]
他给出了一个操作化的判断公式：对方的话让你感到不舒服时，问自己"他在交换什么"。
这个公式的具体应用见 [这些案例].
[本段基于: SRC-LDM-045, 18:30-20:00, "[原话: 你只要问自己——他在交换什么——你就知道了]"]
```

### 1.4 段落级溯源覆盖率标准

| 文档类型 | 最低覆盖率 | 说明 |
|---------|---------|------|
| CON（概念） | ≥80%段落有溯源 | CON定义必须高度锚定原文 |
| OPI（观点） | ≥90%段落有溯源 | OPI是此人的观点，每句都要可查 |
| CAS（案例） | ≥80%段落有溯源 | 事件描述+此人分析都要溯源 |
| SOL（方案） | ≥50%段落有溯源 | 方案可能是分析者综合的（允许≤50%无溯源） |
| 主题地图 | ≥30%段落有溯源 | 主题地图主要是索引和综合，溯源比例可较低 |

## Part 2: 全量自动化双链验证（补缺口#12）

### 2.1 v2.0 → v3.0 的关键变化

| 检查项 | v2.0 (Step 5.3) | v3.0 (Step 5.3 升级) |
|--------|----------------|---------------------|
| 死链检查 | 未说明是全量还是抽查 | **全量**扫描所有 `[[链接]]` |
| 素材覆盖率 | 未说明 | **全量**统计所有SRC-ID的使用情况 |
| 结论可反查 | 随机抽查5条 | **全量**扫描+"随机抽查5条人工验证" |
| 新增: 段落级溯源覆盖率 | 无 | **全量**扫描每个文件的段落溯源比例 |

### 2.2 全量检查项

```markdown
## 5.3 双链完整性终检（v3.0 升级版）

### 自动化全量检查（100%覆盖，不允许抽查）

#### 检查1: 死链全量扫描
- 扫描范围: 产物目录下所有 .md 文件
- 扫描目标: 所有 `[[链接]]` 和 YAML relationships 中的 target
- 验证: 每个链接的目标ID/文件是否存在
- 结果: 死链数 = N (必须 = 0)
- 不通过 → 🚫 禁止发布

#### 检查2: 素材覆盖率全量统计
- 统计: 所有SRC-ID在产物中至少被引用1次的数量
- 计算: 被引用的SRC-ID数 / 注册表总SRC-ID数 = X%
- 目标: ≥70%
- 未覆盖的SRC-ID: [列表 + 每条为什么未被引用]

#### 检查3: 段落级溯源覆盖率全量统计
- 扫描: 每个 .md 文件的每个段落
- 统计: 有溯源标记的段落数 / 总段落数 = X%
- 按文件类型分别统计:
  | 文件类型 | 覆盖率 | 标准 | 通过？ |
  |---------|--------|------|--------|
  | CON | X% | ≥80% | ✅/❌ |
  | OPI | X% | ≥90% | ✅/❌ |
  | CAS | X% | ≥80% | ✅/❌ |
  | SOL | X% | ≥50% | ✅/❌ |
  | 主题地图 | X% | ≥30% | ✅/❌ |

#### 检查4: 孤儿单元全量扫描
- 统计: 未被任何其他单元引用的单元数
- 标准: 孤儿数 ≤ 总单元数的5%
- 孤儿列表: [列出所有孤儿 + 是否需要删除/补链接]

#### 检查5: 双向性全量检查
- 扫描: 所有 YAML relationships
- 检查: 如果A的relationships包含B，B是否也包含A
- 不对称关系: [列出]

### 随机深度验证（抽查5条，人工执行）

从以上全量检查通过后，随机抽取：
1. 5条SRC-ID → 验证其原文确实支持引用的结论
2. 5个关键结论 → 追溯完整溯源链（结论→OPI→SRC→原文位置）

### 发布判定
- 全量死链=0 + 素材覆盖率≥70% + 段落级溯源达标 + 孤儿≤5% + 双向性闭合
- → ✅ 可以发布 / ❌ 需要修复（列出具体修复项）
```

## Part 3: 动态双链更新机制（补缺口#13）

### 3.1 活人蒸馏的动态更新协议

当蒸馏对象是活人且持续产出新内容时，执行以下协议：

```
动态更新周期: 每季度 或 新内容积累≥10条新素材 时触发

Step U0: 新素材注册
  - 为新素材分配SRC-ID（续编）
  - 追加到素材注册表
  - 更新"素材完整性声明"

Step U1: 新素材扫描
  - 扫描新素材的内容方向是否与已有方向一致
  - 内容方向有变化 → 更新 Step 0.15
  - 新素材是否暴露了之前未覆盖的领域 → 更新覆盖面积

Step U2: 已有单元更新
  对每个已有单元，检查新素材是否:
  - 支撑已有结论 → 追加 source_documents (不修改核心内容)
  - 补充已有概念 → 追加典型应用
  - 与已有观点冲突 → 标注冲突 + 进入纠错流程
  - 提供了更高精度的引用 → 升级引用级别（L1→L2, L2→L3）

Step U3: 新单元创建
  - 新素材中是否有新的CON/OPI/CAS/SOL → 创建新单元
  - 即时建立关系（Step 1.9a）

Step U4: 关系更新
  - 新素材是否建立了新关系 → 追加到关系网络
  - 新素材是否暴露了之前未发现的冲突 → 追加到内在张力
  - 新素材是否改变了某个关系的强度 → 调整关系强度

Step U5: 双链重检
  - 重新运行 Step 2.9b（死链检查）
  - 新增单元的双链完整性

Step U6: 版本标记
  - 被更新的单元 version += 1
  - 旧版本不删除，保存在 _history/ 目录
  - 更新日志追加到 CHANGELOG.md
```

### 3.2 版本管理标准

```yaml
---
id: CON-001
version: 3
version_history:
  - version: 1
    date: 2024-06-01
    changes: "初始创建，基于 SRC-LDM-001 ~ SRC-LDM-450"
  - version: 2
    date: 2024-09-15
    changes: "新增SRC-LDM-451~500的验证案例，升级1个典型应用"
  - version: 3
    date: 2025-01-10
    changes: "新素材揭示了此概念的失效条件（SRC-LDM-601），补充失效条件"
    breaking: false
---
```

### 3.3 动态更新时的诚实边界更新

```
每次动态更新后，诚实边界自动检查:
- 新增素材是否填补了之前的缺口 → 诚实边界缩减
- 新增素材是否暴露了新的缺口 → 诚实边界扩大
- 新增素材是否削弱了之前的结论 → 标注弱化的结论
```

## Part 4: 链接强度分级（补缺口#14）

### 4.1 三级链接强度

每条关系必须标注强度，在YAML relationships中新增 `strength` 字段：

```yaml
relationships:
  - type: 证明
    target: OPI-003
    strength: ★★★  # 强
    basis: "此人在SRC-XX-005中明确用此案例证明此观点"
  - type: 解释
    target: OPI-007
    strength: ★★☆  # 中
    basis: "此人未明确说A解释B，但A的框架应用于B场景时逻辑自洽"
  - type: 共享主题
    target: OPI-012
    strength: ★☆☆  # 弱
    basis: "讨论同一主题但从不同角度，彼此无直接逻辑关系"
```

### 4.2 强度判断标准

| 强度 | 含义 | 判断标准 |
|------|------|---------|
| ★★★ 强 | 此人有明确表述的连接 | 素材中有"A因为B""A证明B""A和B的关系是"等表述 |
| ★★☆ 中 | 分析者推断但逻辑自洽的连接 | 此人未明确说，但A和B的逻辑关系不需要此人的额外声明就能成立 |
| ★☆☆ 弱 | 松散的、共享主题的连接 | A和B讨论同一话题但方向不同、角度不同、彼此独立 |

### 4.3 强度对网络分析的影响

```
网络中心计算:
  中心度 = Σ (关系强度权重)
  其中: ★★★ = 3, ★★☆ = 2, ★☆☆ = 1

结构洞检测:
  只有 ★★★ 和 ★★☆ 关系的缺失才算"结构洞"
  ★☆☆（共享主题）的缺失不算结构洞——两个概念可以共享主题但没有强逻辑关系

交叉分析权重:
  在Phase 4交叉分析中，只有 ★★★ 和 ★★☆ 关系参与深度交叉
  ★☆☆ 关系仅用于背景标注
```

### 4.4 强度分布健康检查

一次蒸馏产物的关系强度分布应该是健康的（不能全是★☆☆）：

| 强度 | 健康占比 | 警告 |
|------|---------|------|
| ★★★ 强 | ≥40% | 如果<20%→可能大量关系缺乏原文支撑 |
| ★★☆ 中 | 30-50% | — |
| ★☆☆ 弱 | ≤30% | 如果>50%→关系网络太松散，可能大量"共享主题"掩盖了真正的结构洞 |

## Part 5: 文档内的语义双链（Obsidian兼容）

### 5.1 文档内链接格式

```
知识库产物标准:
- 内容单元间引用: [[CON-001]]
- 带别名的引用: [[CON-001|价值交换]]
- 段落级引用: [[CON-001#段3|操作公式部分]]
- 主题地图引用: [[05-主题地图/核心操作方法编目]]
- 素材引用: [[01-原始素材区/SRC-XX-003|2024-03-15 连麦对话]]
- 跨文件夹引用: [[02-内容单元库/CON/CON-001]]
```

### 5.2 语义双链 — 不只是"链接存在"，还要说明"为什么链接"

在 .md 正文中，每个 `[[链接]]` 应附带一句话说明为什么链接：

```markdown
✅ 好: "此观点被 [[CON-001|价值交换]] 解释——因为对方的行为模式可以用价值交换公式直接计算"
❌ 差: "相关概念: [[CON-001]]"（没有说明为什么相关）
```

## 一句话总结

**v2.0的双链是"文档级死链检查+随机抽查"。v3.0的双链是"段落级溯源+全量自动化验证+动态更新协议+强度分级"。溯源从单元到句子，验证从抽查到全量，维护从一次性的到持续的。**
