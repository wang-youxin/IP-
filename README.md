| 用户说的 | 含义 | 对应的框架步骤 | 打包产出 |
|---------|------|--------------|---------|

| **第二次蒸馏** | 特点之间关联 | Phase -1.5权重 + Phase -1.6星级链接 + Phase 3对照 + Step 3.5对抗验证 + Phase 4交叉分析 + Step 4.4动态双链 → **Step 4.5打包** | **`Round-2-Report.md`** |

```

# Part -1: 🆕 前置阶段 — 语义切分+场景环境+权重计算+星级链接（v3.4~v3.6新增）

> ⚠️ **v3.4重大架构升级**：在所有分析之前，新增三个前置步骤。原始素材 → 语义切分为场景单元 → 数据驱动权重计算 → 星级评定与全量链接 → 进入Phase 0。**这两个步骤决定了后续所有蒸馏的质量上限。**

## -1.1 Phase -1: 语义切分与场景环境分析（v3.4）

**为什么需要**：原始素材是"一坨"——对话、文章、视频混在一起，时间线、场景、环境全混。直接分析会导致上下文丢失、环境因素被忽略、跨场景模式无法检测。

**做什么**：
1. **语义切分**：按5条规则（场景变换/话题切换/对话回合/情绪突变/叙述结构）将素材切为独立场景单元
2. **场景文档化**：每个场景生成标准化文档（YAML frontmatter + 场景摘要 + 六维环境分析）
3. **场景关系建立**：6种关系类型（时序/因果/情绪演化/主题延续/结构对应/独立）
4. **场景地图**：输出全景场景关系图

**输出物**：N个场景文档 + 场景地图 + 场景索引。详见 `Phase--1-语义切分与场景环境分析.md`

## -1.2 Phase -1.5: 数据驱动权重计算（v3.5）

**为什么需要**：传统的关键词权重靠"专家判断"或"感觉重要"。但在大量数据面前，权重应该从数据中涌现——就像Transformer的Attention机制，不是预设的，而是算出来的。

**做什么**（类Transformer 7Head Multi-Head Attention）：
1. **Head 1: TF-W（词频权重）** — 基础统计，词在此人素材中的出现频率
2. **Head 2: CO-W（共现权重）** 🔥核心 — 共现网络中心性（度中心性/介数中心性/特征向量中心性）
3. **Head 3: POS-W（位置权重）** — 出现在关键位置（开头/结尾/转折处）的加权
4. **Head 4: CSC-W（跨场景一致性权重）** — 跨场景稳定出现→高权重，仅单场景→低权重
5. **Head 5: EI-W（情绪强度权重）** — 高情绪场景中出现的关键词加权
6. **Head 6: RT-W（言行比权重）** — 说的≠做的→降低权重修正
7. **Head 7: STV-W（主动触发权重）** — 此人主动提及 vs 被问才答

**聚合公式**：`综合权重 = 0.25×CO-W + 0.20×CSC-W + 0.15×STV-W + 0.10×TF-W + 0.10×POS-W + 0.10×RT-W + 0.10×EI-W`

**输出物**：关键词权重矩阵 + 共现网络图 + 隐藏概念列表 + 维度激活建议。详见 `Phase--1.5-数据驱动权重计算.md`

## -1.3 Phase -1.6: 星级权重与全量链接标准（v3.6）

**为什么需要**：分析出几百个思维点后，哪些最重要？每个思维跟哪些案例挂钩？有没有孤儿思维或死链接？这需要一个系统性的评级和链接验证体系。

**做什么**：
1. **四维星级评定**：`综合得分 = 0.40×思维层级 + 0.30×案例支撑 + 0.20×跨场景一致性 + 0.10×独特性`
   - ≥0.90 → ⭐⭐⭐⭐⭐，≥0.70 → ⭐⭐⭐⭐，≥0.50 → ⭐⭐⭐，≥0.30 → ⭐⭐，<0.30 → ⭐
2. **思维-案例链接矩阵**：每个思维↔案例双向链接，链接强度★★★/★★☆/★☆☆三级，链接密度自动计算
3. **100%全量铁律**：4项硬性门禁——素材覆盖率=100%、双向链接率=100%、死链=0、孤儿=0。一项不达标→产物不可发布

**输出物**：星级总览 + 链接矩阵 + 蒸馏完整度仪表盘。详见 `Phase--1.6-星级权重与全量链接标准.md`

> 📖 **详细内容请直接阅读这三份文档**，每份都包含完整的提示词模板和可执行步骤。

## 0.1 三色编码系统

本框架使用三色编码，从设计原理上根本不同于"模板套用"型蒸馏：

# Part 1: Phase 0 — 素材诊断（核心入口）

> ⚠️ **这是整个框架最重要的部分。Phase 0 的判断质量 = 后续所有蒸馏的高度和深度上限。在这里多花1小时，后面少走10小时弯路。**

## 1.1 诊断步骤总览

```
Step 0.0: 素材注册 ─── 给每条素材分配唯一SRC-ID + 保存原文副本（溯源基础）
    ↓
Step 0.1: 素材清点 ─── 有什么？什么类型？多少量？
    ↓
Step 0.15: 内容方向与覆盖面积判定 ─── 🆕v3.0 此人的内容在谈什么？覆盖了哪些领域？
    ↓
Step 0.2: 素材质量评级 ─── 一手还是二手？时间跨度？覆盖面？
    ↓
Step 0.3: 人物类型判定 ─── 这个人是干什么的？他的核心活动是什么？
    ↓
Step 0.4: 运行🔴红色判断规则 ─── 输出维度激活清单
```

> 🆕 **v3.0新增 Step 0.0 和 Step 0.15**：
> - **Step 0.0** 是溯源的物理基础——没有SRC-ID，后续所有溯源要求都是空谈。
> - **Step 0.15** 是蒸馏的方向盘——内容方向和职业类型可能不同（鲁大魔职业是自媒体人，但90%内容是诊断商业问题）。在评级之前先搞清楚方向，决定了后续所有维度的权重分配。

> 详见 `v3.0-Step0.15-内容方向与覆盖面积判定.md`

## 1.2 Step 0.0: 素材注册（溯源基础）

> ⚠️ **在任何分析之前，必须先建立素材注册表。没有溯源编号的蒸馏 = 不可验证的推测。**

### 注册格式

每条素材分配唯一ID：`SRC-[人物缩写]-[序号]`

```markdown
## 素材注册表

| SRC-ID | 日期 | 类型 | 标题/描述 | 原文位置 | 字数/时长 |
|--------|------|------|----------|---------|----------|
| SRC-XX-001 | YYYY-MM-DD | 对话/著作/... | [描述] | [文件路径/URL] | [量] |
| SRC-XX-002 | ... | ... | ... | ... | ... |

总计: N条素材
```

### 原文保存铁律
- 每一条SRC-ID对应的原文必须保存为独立文件或独立记录
- 原文不可修改、不可概括、不可"优化表述"
- 保存格式: 纯文本（优先）或 Markdown（保留结构）
- 禁止行为: 在原文中插入分析者评论、用"..."跳过"不重要"部分但不标注、意译原文而不保留原句

### 完整性声明
```markdown
## 素材完整性声明
- 总素材数: N条
- 时间跨度: YYYY-MM-DD 至 YYYY-MM-DD
- 素材类型分布: [列出类型及数量]
- 已知缺失: [列出知道存在但未获取的素材]
- 诚实声明: 此蒸馏基于上述N条素材。缺失的素材类型可能影响以下维度的准确性: [列出]
```

## 1.4 Step 0.15: 内容方向与覆盖面积判定（🆕v3.0）

> **插入位置**：Step 0.1（素材清点）和 Step 0.2（素材质量评级）之间
> **目的**：素材类型相同，但内容方向不同 → 蒸馏策略完全不同。在评级之前先搞清楚方向。
> **核心理念**：第一步决定了后续的高度和深度。

### 三个核心问题

1. **内容方向是什么？** — 此人的内容输出主要在谈论什么？按主题聚类，用占比说话。
   - 不是问"他是什么职业"，是问"他说的话主要在讲什么"
   - 鲁大魔的职业是"自媒体人"，但90%的内容方向是"诊断别人的商业问题"

2. **覆盖的面积是什么？** — 覆盖了哪些主题领域？深度如何？有没有"该谈但没谈"的领域？

3. **实际素材能支撑什么？** — 基于现有素材，哪些蒸馏方向可执行？哪些素材不足？

### 内容方向聚类规则

| 占比 | 分类 | 策略 |
|------|------|------|
| ≥30% | **核心方向** | 后续蒸馏必须覆盖 |
| 10-30% | **次要方向** | 选2-3个最有特色的做 |
| <10% | **边缘方向** | 特有维度触发则做，否则不强制 |

### 素材支撑度评估

| 支撑度 | 条件 | 行动 |
|--------|------|------|
| 🟢 充足 | 素材类型+数量满足蒸馏需求 | ✅ 可执行 |
| 🟡 勉强 | 素材有但类型单一或数量偏少 | ⚠️ 可执行但置信度降级 |
| 🔴 不足 | 素材缺失或仅二手来源 | ❌ → 进入诚实边界 |

> 📄 完整规范见 `v3.0-Step0.15-内容方向与覆盖面积判定.md`

## 1.5 Step 0.2: 素材质量评级

### 一手占比评级

| 一手占比 | 评级 | 蒸馏深度上限 |
|---------|------|------------|
| >70% | 🟢 优秀 | 可做心智模型级别蒸馏（三重验证） |
| 40-70% | 🟡 良好 | 可做概念+观点级别蒸馏 |
| <40% | 🟠 受限 | 诚实边界扩大，所有结论标注推测比例 |

### 时间跨度评级

| 时间跨度 | 评级 | 蒸馏产出 |
|---------|------|---------|
| >10年 | 🟢 完整 | 完整思想演进图 + 转折点分析 |
| 3-10年 | 🟡 足够 | 关键转折点 + 当前快照 |
| 1-3年 | 🟠 快照 | 仅当前状态，标注"未覆盖思想演进" |
| <1年 | 🔴 片段 | 极度受限，诚实边界必须大篇幅说明 |

### 覆盖面评级

| 覆盖面 | 判断标准 | 评级 |
|--------|---------|------|
| 全面 | 著作+对话+社交媒体+决策记录 至少覆盖3类 | 🟢 |
| 中等 | 覆盖2-3类 | 🟡 |
| 片面 | 只有1类 | 🟠 |

> ⚠️ **单源偏差警告**：如果所有素材来自同一类型（如只有社交媒体），蒸馏结果的偏差风险极高。必须在诚实边界中大幅标注。

## 1.6 Step 0.3: 人物类型判定

### 8种基础类型 + 复合型

| 类型 | 核心活动 | 典型人物 | 判断信号 |
|------|---------|---------|---------|
| **诊断型** | 解决别人的问题 | 鲁大魔、咨询顾问、心理治疗师 | 大量QA对话、回应模式、方法论输出 |
| **创作型** | 创造作品 | 作家、导演、画家、音乐人 | 有作品集、谈论创作过程、审美观点 |
| **产品型** | 设计和构建产品 | Steve Jobs、张小龙、产品经理 | 产品发布、设计哲学、用户体验论述 |
| **学术型** | 研究和教学 | 费曼、科学家、学者 | 论文、课程、学术辩论、研究方法论 |
| **投资型** | 资本配置和判断 | 巴菲特、Naval、Taleb | 投资论述、市场分析、风险评估框架 |
| **哲学型** | 思考抽象问题 | 芒格、哲学家、思想者 | 核心命题、论证结构、思想体系 |
| **表演型** | 在观众面前表演/展示 | MrBeast、喜剧演员、演讲者 | 舞台方法论、观众心理学、节奏控制 |

### 复合型判定

大多数人属于**复合型**。判定规则：
- 选出**主要类型**（占50%+内容输出的活动）
- 选出**次要类型**（占20-40%的活动）
- 主类型决定核心插件，次类型添加补充维度

**示例**：
- 鲁大魔 = 诊断型(70%) + 表演型(20%) + 哲学型(10%)

- Jobs = 产品型(60%) + 表演型(25%) + 管理型(15%)
- 费曼 = 学术型(50%) + 诊断型/教学(30%) + 表演型(20%)

## Step 0.0: 素材注册（溯源基础）

> ⚠️ **在一切分析之前，必须先建立素材注册表。没有溯源编号的蒸馏=不可验证的推测。**

### 为什么这一步必须放在最前面

鲁大魔库的核心经验：904条原始素材，每条有唯一编号（LD-001~LD-904）。978个内容单元，每个单元的 `source_documents` 字段指向具体的素材编号。797/904条被引用。这使得：
- 任何结论都可以追溯到原始素材中的具体位置
- 任何观点都可以被第三方验证
- 任何错误都可以被定位和修正

**没有素材注册的蒸馏=不可验证的推测。**

### 执行步骤

1. 为每条素材分配唯一ID：`SRC-[人物缩写]-[序号]`
2. 保存原始素材副本（原文/transcript/导出）
3. 建立来源注册表

### 输出格式

```markdown
## 素材注册表 — [人物名]

| 素材ID | 类型 | 日期 | 标题/描述 | 字数/时长 | 文件位置 | 备注 |
|--------|------|------|----------|----------|---------|------|
| SRC-XX-001 | 直播录屏 | 2024-03-15 | 连麦: 关于XX问题的讨论 | 45min | 01-原始素材区/ | |
| SRC-XX-002 | 本人著作 | 2020 | 《XX》第三章 | 5000字 | 01-原始素材区/ | |
| ... | ... | ... | ... | ... | ... | ... |

总计: N条素材
```

### 溯源铁律（贯穿全部Phase）

```
1. 每个CON/OPI/CAS/SOL的YAML frontmatter必须包含 source_documents 字段
2. 每个关键结论的正文必须包含原文引用（格式: "SRC-XX-001: [原文片段]"）
3. 推测性内容必须标注"推测"并说明推测依据
4. 无法溯源的观点必须进入诚实边界
```

## 2.1 表达DNA

### 目的
提取此人的语言特征——不是"他应该怎么说话"，而是"他实际上怎么说话"。这些特征是角色扮演型AI分身的核心输入。

### 6个子维度

| 子维度 | 核心问题 | 提取方法 |
|--------|---------|---------|
| **句式偏好** | 他怎么组织句子？ | 统计平均句长、句式结构分布、段落组织方式 |
| **词汇特征** | 他用什么词？ | 高频词Top 20、自创术语、禁忌词、口头禅 |
| **节奏感** | 他说话的信息密度和停顿模式？ | 观点密度（观点数/千字）、停顿位置分析 |
| **幽默方式** | 他怎么让人笑？ | 讽刺/自嘲/荒诞对比/冷幽默/预期违背/不幽默 |
| **确定性表达** | 他对自己的判断有多确定？ | hedging language频率、面对不确定问题的应对 |
| **引用习惯** | 他引用谁？怎么引用？ | 引用密度、引用来源分布、引用方式分类 |

### 强制溯源要求

每个子维度的分析必须配原文例句，格式：
```
例句: "[原文]" — SRC-XX-003, 12:34
```

高频词统计必须提供原始计数数据，不能只有结论。

### 输出格式

```markdown
## 表达DNA — [人物名]

### 句式偏好
- 平均句长: [N]字
- 句式分布: 陈述X%/反问Y%/对称Z%
- 典型例句:
  - "[原文]" — SRC-XX-003 (短句示例)
  - "[原文]" — SRC-XX-007 (长句示例)

### 词汇特征
- 高频词Top 20: [词(次数), ...]
- 自创术语: [词] = "[此人定义]" — 首次出现: SRC-XX-001
- 禁忌词: [词] — 证据: 此人明确说过"我不用X这个词" — SRC-XX-005
- 口头禅: [词] — 出现频率: N次/万字

### 节奏感
- 信息密度: [N]个独立观点/千字
- 先结论/先铺垫: [判定+证据]
- 停顿位置: [分析+标注示例]

### 幽默方式
- 类型: [判定]
- 实例:
  1. "[原文]" — SRC-XX-008
  2. "[原文]" — SRC-XX-012

### 确定性表达
- 类型: [我不确定型/很明显型/只有这样型]
- hedging language使用频率: [N次/万字]
- 面对不确定问题: "[此人典型回应]" — SRC-XX-015

### 引用习惯
- 引用密度: [N次/千字]
- 最常引用: [人名/书名](N次)
- 引用方式: 直接引述X%/内化重述Y%/不提来源Z%

### 一句话DNA
[用一句话描述此人表达特征——要能"读100字认出是谁"]
```

### 质量检查（v3.0标准）
- [ ] 6个子维度全部覆盖
- [ ] 每个子维度有≥10条原文例句（带SRC-ID+时间戳）← v3.0升级
- [ ] 百分比+原始计数双标注（如"73% (47/64次)"）
- [ ] 段落级溯源覆盖率 ≥80%（每个关键论断有SRC位置）
- [ ] 一句话DNA是否可区分此人和其他14个参考人物？

## 2.5 内在张力/矛盾

### 目的

|------|------|------|

| **D类: 场景依赖** | 同一原则在不同场景应用方式相反 | 对内部团队vs对外部客户的同一原则 |

## 2.6 概念体系

### 目的
提取此人自创或独特使用的核心概念，形式化定义，建立概念层级。

### 6层概念层级（来自鲁大魔库）

| 层级 | 说明 | 判断标准 |
|------|------|---------|
| **公理层** | 永远运行，不依赖其他概念 | 去掉它，整个体系崩塌 |
| **操作层** | 具体的认知操作/思维动作 | 有明确的操作公式 |
| **身份层** | 关于"我是谁""我该怎么定位" | 与自我认知和边界相关 |
| **传播层** | 关于如何表达和传播 | 与媒介/表达/受众相关 |
| **业务层** | 关于具体领域的方法论 | 与特定领域绑定 |
| **策略层** | 关于长期方向和取舍 | 有时效性和场景性 |

### 概念提取规则

**是CON（概念）**：这个词的定义变了→整个推理链改变

## 2.9 关系网络

> ⚠️ **这一步拆为两个子步骤：2.9a 关系构建（每个单元创建时即时建立）+ 2.9b 双链验证门禁（全部单元完成后死链检查）。死链未清零 = 不允许进入Phase 4。**

### 2.9a: 关系构建

#### 5种关系类型

| 关系类型 | 含义 | 示例 | 何时建立 |
|---------|------|------|---------|
| **回应** | A是针对问题/提问的回答 | OPI-001 回应 提问X | OPI创建时 |

## Step 2.0: 重复行为模式扫描

### 目的
扫描全部素材，找出此人**反复出现的行为模式**。不问"这属于什么类型"，只问"他在反复做什么"。

### 扫描规则

```
逐条扫描素材中的每一个"此人做了什么"的片段:
1. 标记重复出现 ≥3次 的行为模式
2. 每个模式记录:
   - 触发条件: 什么事情发生之后，他做了这个行为？
   - 行为本身: 他具体做了什么？（用原文描述，不要概括）
   - 产出结果: 这个行为导致了什么？
   - 出现次数和SRC-ID列表
3. 不急于命名——先准确描述
4. 不筛选——即使看起来"不是方法论"的重复行为也记录
   （鲁大魔的"打断对方"最初看起来只是习惯，后来成了核心方法论）

### 行为模式的类型（帮助扫描，不限制）

| 模式类型 | 定义 | 示例（鲁大魔） |
|---------|------|-------------|
| **干预型** | 面对外部输入时的固定反应 | 面对提问不直接回答，反向拆解提问本身 |
| **产出型** | 创造/生产东西时的固定流程 | 写内容时先列框架→填案例→砍掉一半 |
| **判断型** | 做选择时的固定标准 | 判断一个人"是否有救"先看自知度 |
| **维护型** | 维持自身状态的固定操作 | 每季度清理一次知识库，删掉过时的 |
| **互动型** | 与人交流时的固定模式 | 连麦时先沉默3秒，然后问"你刚才说的X是什么意思" |

## Step 2.1: 模式聚类与命名

### 目的
将Step 2.0的原始模式分组，按**本质相似度**聚类。给每个聚类命名——优先使用此人自己的语言。

### 聚类规则

```
聚类不是按"话题"（讨论什么），是按"结构"（怎么做的）:

✅ 正确聚类:
  - 模式A（面对提问反向拆解）+ 模式B（面对质疑不辩护，反向追问）→ 聚类为"不进入对方框架"
  - 两者结构相同（都不在对方设定的层面回应），只是场景不同

❌ 错误聚类:
  - 模式A（讨论管理时的提问方式）+ 模式B（讨论投资时的提问方式）→ 不应仅因话题都是"提问"就聚在一起
  - 结构不同的行为不能强行聚类
```

### 命名规则

```
优先级:
1. 此人自己命名了这个模式 → 用他的命名（鲁大魔叫"破锁"，就用"破锁"）
2. 此人没命名但反复用同一个词描述 → 用那个词
3. 以上都没有 → 用描述性命名（"[动作]+[对象]"格式，如"反向拆解提问"）

禁止: 用分析者的术语强行命名（不要叫"苏格拉底式反问"除非此人自己这么说）
```

### 输出格式

```markdown

**本质定义**: [这个聚类中所有模式共享的底层结构]

**包含的原始模式**:
| 原始模式 | 场景 | 出现次数 | 变异特征 |
|---------|------|---------|---------|

## Step 2.2: 模式形式化

### 目的

### 形式化标准（v3.0统计验证标准）

每个模式必须通过以下验证：

| 验证项 | 标准 | 不达标则 |

| **案例数** | ≥5个验证案例（带SRC-ID） | 标注"案例不足，暂定" |
| **反例数** | ≥2个反例分析 | 标注"反例不完整" |
| **推导链** | 完整的"如果X→因为Y→所以Z"链 | 标注"推导链不完整" |

## Step 2.3: 模式层级化

### 目的
将形式化的模式按**依赖层级**组织。参照鲁大魔库的4层架构作为组织参考，但不强制套用。

### 四层架构（组织框架，非强制模板）

| 层级 | 定义 | 判断标准 | 鲁大魔示例 |
|------|------|---------|----------|
| **公理层** | 永远运行，不依赖其他模式 | 去掉它，整个操作体系崩塌 | "价值交换"(所有其他武器都基于此) |
| **元规则层** | 跨场景的稳定规则，决定"何时用哪个操作级模式" | 不局限于单一场景 | "不对称回应律"、"自知锚定先行律" |
| **操作层** | 具体场景的可执行操作 | 有明确的触发条件+操作步骤 | "破锁"、"归因翻转"、"变量压缩" |
| **证据层** | 支撑上三层的具体案例/验证数据 | 回答了"你怎么知道的" | 每个武器的验证案例 |

### 层级判断规则

```
1. 公理层判断: 假设"此人不用这个模式了"→他的其他模式是否还能成立？
   如果不能 → 公理层。如果能 → 不是公理层。

2. 元规则层判断: 此模式是否决定了"什么时候用操作层的哪个模式"？
   如果是 → 元规则层。如果是操作本身 → 操作层。

3. 操作层判断: 此模式是否有明确的触发条件+操作步骤？
   如果是 → 操作层。如果只是统计事实没有步骤 → 可能是行为倾向，降级到L0。

4. 证据层: 不是模式，是模式的数据支撑。如果Step 2.2的反例/案例足够丰富 → 自动形成证据层。
```

### 输出格式

```markdown
## 操作模式层级 — [人物名]

### 层级总览
| 层级 | 模式数 | L0 | L1 | L2 | L3 |
|------|--------|-----|-----|-----|-----|
| 公理层 | N | — | M | — | — |
| 元规则层 | N | — | M | — | — |
| 操作层 | N | M | M | — | — |
| 证据层 | — | — | — | — | — |

(L0-L3深度待Phase 3对照后确定)

### 层级图
```
公理层: [模式A], [模式B]
  ↓ 决定/约束
元规则层: [模式C], [模式D]
  ↓ 调度/选择
操作层: [模式E], [模式F], [模式G], [模式H]
  ↓ 被验证
证据层: CAS-001~CAS-020
```

### 层级归属理由
| 模式 | 层级 | 理由 |
|------|------|------|
| [模式A] | 公理 | 如果去掉它，[模式C/D/E/F]都不再成立——因为[...] |
| [模式C] | 元规则 | 它决定了"在场景X中用操作E还是操作F"——因为[...] |
| ... | ... | ... |

### 与鲁大魔库层级结构的对比
| 鲁大魔库 | 此人 |
|---------|------|
| 公理层: 1个(价值交换) | 公理层: [N]个 — [异同分析] |
| 元规则层: 12条 | 元规则层: [N]条 — [异同分析] |
| 操作层: 17种武器 | 操作层: [N]个 — [异同分析] |
```

### 质量检查
- [ ] 每个模式的层级归属有理由（不是"感觉"）
- [ ] 公理层判断通过了"去掉它"测试
- [ ] 与鲁大魔库的层级对比已完成

|---------|------|---------|

| 产品型但不用用户研究 | — | 产品非标模式 |

**核心逻辑**：此人不用常规方法却能出成果→他有替代机制。这个替代机制往往是他最独特的地方。

**执行**：Phase 2 Step 2.0 扫描时自动标记"该类型常规做法此人未做"的领域→Step 2.1聚类时作为独立聚类分析。

## 🆕 规则R8: 表达与思维关系判定（v3.2新增）

> **来源**：费曼案例暴露的问题——费曼的类比不是"教学技巧"，是他的思维本身。

判断此人的表达方式是"修辞工具"还是"思维本身的一部分"：

| | 修辞工具型 | 思维一体型 |
|---|----------|----------|
| **本质** | 表达服务于沟通——先想清楚，再用技巧传递 | 表达即思维——在表达的过程中完成思考 |
| **即兴vs准备** | 准备稿和即兴风格相似 | 即兴表达和自己写的有显著差异 |
| **抽象→具象** | 先有抽象结论，再加例子 | 从具体例子跳跃到抽象，无中间层 |
| **自述** | "我只是把想好的说出来" | "我不知道我怎么想的，直到我说出来" |

**激活条件**：当素材同时满足①有即兴+准备两种素材 ②表达DNA显示即兴和准备风格显著不同 ③此人谈论过自己的思维过程。
**执行位置**：Phase 4 交叉分析——表达DNA × 智识谱系 × 时间线三重交叉。

### D2: 诊断引擎

## 附录F: 分块执行协议（v3.3新增）

> **解决的问题**：鲁大魔904条对话≈几百万token，费曼2本自传+3卷讲义+8段访谈≈几百万token。任何真实人物的蒸馏都远超单次上下文窗口。框架的提示词假设"扫描全部素材"——分块协议给出分批执行的工程方法。

### F.1 触发条件

| 素材规模 | 策略 |
|---------|------|
| ≤200条素材 且 ≤50万字 | 🟢 不分块——可尝试一次性执行 |
| 200-500条 或 50-150万字 | 🟡 轻量分块——按素材类型分批，2-3批 |
| >500条 或 >150万字 | 🔴 完整分块——按内容方向+时间分批，N批 |

### F.2 分块策略

#### 策略A: 按内容方向分块（首选）

基于Step 0.15的内容方向聚类结果：

```
块1: 核心方向(≥30%)的素材 — 优先处理，深度扫描
块2: 次要方向A(10-30%)的素材
块3: 次要方向B(10-30%)的素材
...
块N: 边缘方向(<10%)的素材 — 仅扫描异常
```

**优点**: 每块内部主题一致，涌现模式信噪比高。
**缺点**: 跨方向的模式可能被分割。

#### 策略B: 按时间分块（有>10年跨度时使用）

```
块1: 早期(Y1-Y5)的素材
块2: 中期(Y6-Y10)的素材
块3: 近期(Y11+)的素材
```

**优点**: 直接产出思想演进。
**缺点**: 同期的不同方向混在一起。

#### 策略C: 按素材类型分块（回退策略）

```
块1: 对话/访谈/直播
块2: 著作/长文
块3: 社交媒体/碎片
块4: 外部评价/二手素材
```

**优点**: 最简单。
**缺点**: 跨类型的模式最难合并。

### F.3 逐块执行流程

每块独立执行 Phase 0-3：

```
块N 输入:
  ├── Step 0.0: 素材注册（块内SRC-ID，加块前缀 B[块号]-SRC-XX-NNN）
  ├── Step 0.15: 内容方向判定（仅块内素材的方向分布）
  ├── Phase 1: 通用维度（仅基于块内素材）
  ├── Phase 2: 涌现维度（块内重复行为扫描→聚类→形式化→层级化）
  └── Phase 3: 仅做Step 3.1鲁大魔对照（参考人物对照待合并后统一做）

块N 输出:
  ├── 块内通用维度报告
  ├── 块内涌现模式（临时编号 B[N]-PAT-001, B[N]-PAT-002...）
  └── 块内鲁大魔对照
```

### F.4 跨块合并

所有块完成后，执行合并：

```
合并Step M1: 通用维度合并
  - 时间线: 拼接所有块的时间线 → 完整时间线
  - 表达DNA: 比较各块的表达特征 → 标注"稳定特征"vs"阶段性特征"
  - 智识谱系: 合并影响源 → 标注"早期影响"vs"后期影响"
  - 概念体系: 合并概念 → 标注概念的首次出现时间和演变
  - 其他维度同理

合并Step M2: 涌现模式跨块匹配
  对每个块内涌现模式，在全部块中搜索：
  
  | 匹配类型 | 定义 | 行动 |
  |---------|------|------|
  | **完全匹配** | 不同块中出现结构相同的模式 | 合并为一个模式，验证案例跨块积累 |
  | **部分匹配** | 相似但不完全相同 | 判断是演变（标注时间）还是变异（保留为子模式） |
  | **块内独有** | 仅在某块中出现 | 检查是否因为其他块缺少触发场景 → 如果是，标注"场景限定" |
  | **跨块冲突** | 块A的模式与块B的模式矛盾 | 进入冲突解决流程 → 可能是思想演变或情境差异 |

合并Step M3: 跨块一致性检查
  检查跨块合并后的矛盾：
  - 块A说此人是X，块B说此人是非X → 标注为演变还是矛盾？
  - 块内的内在张力 vs 跨块的内在张力 → 跨块的更可能是真正的矛盾

合并Step M4: 统一对照
  - Step 3.2: 用合并后的涌现模式做14人物库对照
  - Step 3.3: 用合并后的全部数据做思想地图定位
  - Step 3.5: 对抗验证（基于合并后的完整数据）
```

### F.5 各Phase的分块适配

| Phase | 是否可分块 | 适配方法 |
|-------|----------|---------|
| Phase 0 | ✅ 天然可分块 | 每块独立做素材清点+质量评级+类型判定。内容方向和覆盖面积在合并时汇总 |
| Phase 1 | ✅ 可分块 | 每块独立分析11个维度。合并时关注跨块一致性（某特征在各块中都出现→稳定；仅在某块出现→阶段性的） |
| Phase 2 | ⚠️ 需小心 | 涌现模式可能跨块——块A出现了3次的模式，块B出现了2次，单独都不够≥3，合并才够。在合并Step M2中做跨块频率累加 |
| Phase 3 | ❌ 不分块 | 对照需要在合并后统一做——用合并后的完整模式数据 |
| Phase 4-5 | ❌ 不分块 | 交叉分析和产物生成需要全部数据 |

### F.6 分块执行的诚实边界

分块执行引入的额外局限：

| 局限 | 说明 | 缓解方法 |
|------|------|---------|
| **跨块模式遗漏** | 跨块出现但单块不够≥3次的模式可能被漏掉 | 合并Step M2做跨块频率累加 |
| **块边界效应** | 分块方式可能人为断开连续的思想脉络 | 至少用2种分块策略交叉验证关键发现 |
| **早期块偏差** | 先分析的块会影响后分析块的期望 | 每块的分析使用相同的提示词，不传递前块的中间结论 |
| **合成谬误** | 各块都正确的结论合并后可能矛盾 | 合并Step M3专门检查跨块矛盾 |

### F.7 分块执行建议

```
1. 首选按内容方向分块（策略A）—— 内容方向决定蒸馏方向，按方向分块让每块信噪比最高
2. 如果时间跨度>10年，在方向分块内再按时间细分—— 确保捕捉方向内的思想演变
3. 合并时至少用2种方式验证关键模式（如方向分块+时间分块交叉验证）
4. 诚实声明中明确写出分块策略和可能遗漏的模式类型
```

## 附录G: 框架自身的诚实边界（v3.8新增 — 修复B7）

> **原则**：框架要求每个蒸馏产物必须有诚实边界声明。框架本身也必须声明自己不能做什么。

### 框架不能做什么

| 局限 | 说明 | 影响 |
|------|------|------|
| **不能替代领域专家** | 框架能结构化和系统化，但不能创造此人不具备的知识。如果素材中没有，框架不会"发明" | 蒸馏深度上限由素材决定 |
| **上下文窗口限制** | 大素材量（>500条或>150万字）必须分块执行。分块可能导致跨块模式遗漏 | 大项目需要多轮分块+合并 |
| **7Head权重依赖概念提取质量** | 权重计算的质量取决于Phase 1/2概念提取的完整性。概念提取遗漏→权重偏差 | 权重结果仅供参考，需人工抽查 |
| **跨人物验证数据不均衡** | 14个参考人物中，鲁大魔数据完整，费曼/Jobs/Musk有轻量数据，其余10人仅类型标签 | L2(≥3同类型)和L3(≥2跨类型)的跨人物验证对多数人物类型无法公正执行 |
| **对抗验证非真独立** | Step 3.5的独立质疑者与框架的其他Step使用同一LLM。真正的独立验证需要不同模型或不同人 | 关键结论建议外部验证 |
| **虚构人物蒸馏未测试** | 框架基于真实人物设计。虚构人物的"素材"没有私下判断/决策记录/情绪波动 | 虚构人物蒸馏需要调整素材质量评级标准 |
| **活人伦理依赖执行者** | 框架提供了伦理检查（Step 0.0.5），但最终执行取决于使用者 | 蒸馏活人必须获得授权或仅使用公开素材 |
| **非通用AI** | 框架是方法论文档+提示词模板，不是可运行的软件。每一步依赖LLM手动执行 | 全流程执行需要大量LLM交互 |

### 什么情况下框架会失败

| 失败模式 | 触发条件 | 预防措施 |
|---------|---------|---------|
| **素材不足导致空洞蒸馏** | 素材<10条或总字数<5万 | 使用极简模式（附录H），仅执行Phase 0+Phase 1前3个维度 |
| **素材类型单一导致偏见** | 素材全部来自同一平台/同一时间段/同一角色状态 | Phase 0质量评级会预警"类型单一"，但无法补救 |
| **LLM幻觉被当成事实** | 提示词中的"对抗验证"不够，LLM可能自己编造"原文引用" | 所有SRC-ID引用必须被 link_checker.py 验证 |
| **分块合并谬误** | 不同块中得出的矛盾结论在合并时未被发现 | 至少用2种分块策略交叉验证 |

| 等级 | 验证方式 | 独立性 | 适用场景 |

### I.2 单人执行L1独立验证的操作步骤

```
1. 完成Phase 2→产出模式形式化报告
2. 将报告中的"方法论文本"拷贝出来（去掉框架内的上下文）

1. 有没有替代解释？给出至少2个
2. 推导链的哪个环节最薄弱？为什么？
3. 有没有频率幻觉？（提到的案例不够5个→标注）

4. 将独立审核结果与Phase 2输出对比
5. 对每个被质疑的环节：

## 附录M: 虚构人物蒸馏模式（v3.9新增 — 修复E2）

> **问题**：框架基于真实人物设计。虚构人物只有"作品中的表现"，没有"私下判断""决策记录""情绪波动"——所有Phase 0的素材类型分类会失效。

### M.1 虚构人物与真实人物的关键差异

| 维度 | 真实人物 | 虚构人物 |
|------|---------|---------|
| 素材来源 | 对话/自传/视频/社交媒体/私下记录 | 作品文本/作者访谈/改编记录 |
| "私下判断" | 有（私下对话、未公开决策） | 无——只有作者写出来的 |
| 情绪真实性 | 真实情绪 | 被作者安排的情绪 |
| 一致性 | 真实人物可能矛盾、不一致 | 取决于作者的一致性 |
| 思想演进 | 真实的、可追溯的时间线 | 可能是作者的演化而非角色的 |
| 决策证据 | 可观察的实际决策 | 只有作者写的决策 |

### M.2 虚构人物蒸馏的特殊规则

```
1. 将"素材"重新定义为"此角色在作品中的所有行为+对话+内心独白"
2. 额外输入: 作者关于此角色的访谈/评论（标注为"作者视角"，降权×0.5）
3. Phase 0 的"素材质量评级"替换为"角色一致性评级":
   - 作者是否始终如一地写这个角色？（作者一致性）
   - 此角色在不同作品中的表现是否一致？（跨作品一致性）
   - 如果是多位作者写的→标注"多作者，角色可能不一致"
4. 不做"思想演进"分析（那是作者的演进，不是角色的）
5. 涌现维度的频率要求从≥60%降低到≥40%（虚构角色的素材量通常更少）
6. 不做跨人物验证（虚构角色库不存在）
7. 产物必须标注"此蒸馏基于虚构角色[角色名]在[作品名]中的表现"
```

### M.3 虚构角色特有的分析维度

| 维度 | 说明 |
|------|------|
| **作者意图 vs 角色表现** | 作者说这个角色是X，但作品中表现的是Y → 分析差距 |
| **角色弧线** | 此角色在故事中的变化轨迹（替代"思想演进"） |
| **角色功能** | 此角色在叙事中的功能（反派/导师/主角/催化剂）如何影响了他的"方法论" |
| **跨作品稳定性** | 如果同一角色出现在多部作品中→比较一致性 |

## 附录N: 多人协作蒸馏冲突解决协议（v3.9新增 — 修复E3）

> **问题**：两个人同时蒸馏同一个人物，可能产出矛盾的CON/OPI/CAS。没有冲突解决协议。

### N.1 冲突类型与解决方案

| 冲突类型 | 示例 | 解决方案 |
|---------|------|---------|
| **事实冲突** | A说"此人偏好短句"，B说"此人偏好长句" | 回原文仲裁——谁的引用更完整、更全面？ |
| **命名冲突** | A命名"变量压缩法"，B命名"要素递减法" | 比较两人各自给出的定义+案例→取更精确的命名，或合并 |
| **评级冲突** | A评为L2，B评为L1 | 回Step 2.2的L0-L3判定标准→第三人仲裁 |
| **边界冲突** | A认为该进诚实边界，B认为有足够证据 | 保守原则：证据不足的情况下，进诚实边界 |
| **遗漏冲突** | A发现了某个模式，B没发现 | 检查B是否漏看→如果B确认漏看，采纳A的发现 |

### N.2 协作流程

```
1. 分派: 按内容方向分块→每人负责不同方向（不要两人蒸馏同一批素材）
2. 独立执行: 每人独立完成Phase 0-2（不交流中间结果）
3. 交换审核: A审核B的Phase 2形式化报告，B审核A的
   审核标准: 
   - 每个模式的频率计算是否准确？
   - 推导链有无断裂？
   - 失效条件是否完备？
4. 冲突仲裁:
   - 事实冲突→回原文
   - 评级冲突→第三人（或同一LLM扮演仲裁者）
   - 不可调和的分歧→标注双方观点，进诚实边界
5. 合并: 一人负责合并Phase 4/5/6
6. 交叉验证: 另一人独立运行 link_checker.py + coverage_scanner.py
```

### O.2 跨场景模式检测器 `cross_scene_detector.py`

```
功能: 检测跨场景的重复行为模式（辅助Phase 2）

### O.3 方法论冲突检测器 `methodology_conflict_checker.py`
